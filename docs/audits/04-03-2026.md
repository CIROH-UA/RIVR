# RIVR Architecture Audit
**Date:** April 3, 2026
**App version:** 1.0.0+5
**Auditor:** Claude Sonnet 4.6
**Rubric:** `docs/rubric.md` (1000-point scale)

---

## Overall Score: 459 / 1000 — Grade D

The app is functional, well-secured at the secrets level, and has a solid caching foundation. However it carries significant architectural debt: no Clean Architecture layering, no production crash reporting, no observability, and major UX gaps. The score is expected for an app at this stage — this audit exists to map the path to an A.

## Category 1 — API & Networking Layer `82 / 160`

### 1.1 Abstracted HTTP client `20 / 25`
`INoaaApiService` interface exists and `http.Client` is injected into `NoaaApiService`, making it mockable. No raw HTTP calls in widgets or providers. Deduction: no interceptor or middleware layer — auth key injection, logging, and error mapping are inline in the service rather than composable.

### 1.2 Timeout configuration `7 / 10`
Three priority tiers are defined:
```dart
// lib/core/services/noaa_api_service.dart
static const Duration _quickTimeout = Duration(seconds: 15);   // overview
static const Duration _normalTimeout = Duration(seconds: 20);  // supplementary
static const Duration _longTimeout = Duration(seconds: 30);    // complete
```
Good practice. Deduction: `.timeout()` applies a wall-clock total timeout, not separate connect/send/receive granularity.

### 1.3 Retry with exponential backoff + jitter `18 / 25`
`_httpGetWithRetry()` retries on `TimeoutException` and HTTP 5xx with increasing delays:
```dart
await Future.delayed(Duration(seconds: attempt + 1)); // 1s, 2s
```
Max retries capped at 2 (configurable). Deduction: backoff is linear (1s, 2s), not true exponential. No jitter — multiple users hitting the same degraded endpoint will all retry in lock-step.

### 1.4 Request cancellation `0 / 15`
No cancellation mechanism. In-flight requests have no way to be cancelled when a widget or provider is disposed. A user rapidly navigating between reaches will leave dangling network calls that update state on dead providers. The `http` package supports this via `http.Client.close()` but it is not implemented.

### 1.5 Centralized error mapping `12 / 20`
`ApiException` exists and `ErrorService` maps Firebase Auth, Firestore, and network exceptions to user-readable messages. However the error model is flat — `ApiException(message: String)` is a single type, not a hierarchy. There is no way for the UI to differentiate a `NetworkError` from a `ParseError` from a `ServerError` without string-matching the message. Domain errors are not defined in a domain layer.

### 1.6 Auth injection `3 / 10`
The NWM API key is read from `config.dart` (not hardcoded ✅) and appended to the URL at the call site. NOAA's open API requires no auth. There is no interceptor pattern — each call that needs authentication must handle it manually. Partial credit for externalizing the key.

### 1.7 Request/response logging `7 / 10`
`AppLogger` wraps `developer.log()` with `debug`, `info`, `warning`, and `error` levels. Logging is used consistently throughout `NoaaApiService`. Deduction: it is unclear whether `AppLogger.debug()` calls are conditionally stripped in release builds — `developer.log()` calls are compiled into release unless gated behind `assert` or a `kDebugMode` check.

### 1.8 Environment-based configuration `5 / 15`
Base URLs live in `lib/core/config.dart` (gitignored). This externalizes values from source code. However there is no multi-environment support — no concept of dev/staging/prod. Switching from a test API endpoint to production requires manually editing `config.dart`.

### 1.9 Connection-aware behavior `6 / 20`
The stale-while-revalidate pattern in `ReachCacheService` means cached data is shown during poor connectivity. However there is no explicit connectivity detection (`connectivity_plus` is not a dependency), no request queue for when the device is offline, and no programmatic signal to the UI about the connection state.

### 1.10 Request deduplication `2 / 5`
`ForecastService._recentResponseCache` (5-minute TTL, 10-entry max) prevents redundant fetches on rapid screen re-visits. This is response-level deduplication, not request-level — concurrent callers for the same reach within the same 5-minute window will still fire multiple in-flight requests before either returns.

### 1.11 Response validation `2 / 5`
`fromJson` and `fromNoaaApi` methods exist on models. However, Firestore documents are accessed via `doc.data()!` (null assertion) and cast directly to `Map<String, dynamic>`. Unexpected API changes or missing fields will produce runtime exceptions rather than typed `ParseError` objects.

## Category 2 — Caching & Data Persistence `78 / 130`

### 2.1 In-memory cache with TTL `17 / 20`
Three in-memory caches exist in `ForecastService`:
```dart
final Map<String, _TimedEntry<ForecastResponse>> _recentResponseCache = {};
final Map<String, _TimedEntry<double?>> _currentFlowCache = {};
final Map<String, _TimedEntry<String>> _flowCategoryCache = {};
```
The `_TimedEntry<T>` helper with `isExpiredAfter(Duration)` is clean. `_recentResponseCache` is bounded at 10 entries. Deduction: `_currentFlowCache` and `_flowCategoryCache` are unbounded — they grow without limit over a long session.

### 2.2 Persistent disk cache `8 / 25`
All persistence is via `SharedPreferences`. Complex `ReachData` objects are JSON-serialized and stored as strings:
```dart
await _prefs!.setString(key, jsonEncode(reachData.toJson()));
```
`SharedPreferences` is appropriate for primitive scalars and small flags. Using it for 800+ line model objects serialized as JSON strings offers no queryability, no schema migration path, and poor performance at scale. A structured local database (Hive, Isar, drift/SQLite) is the correct tool for this data.

### 2.3 Cache invalidation strategy `14 / 20`
The reach cache strategy is explicit and well-reasoned:
```dart
static const Duration _cacheMaxAge = Duration(days: 180);   // hard eviction
static const Duration _cacheFreshness = Duration(hours: 6); // soft staleness
```
The `cleanupStaleEntries()` method removes expired entries. Deduction: this strategy only covers `ReachData`. Other cached data types (user settings, flow categories) have no documented invalidation strategy.

### 2.4 Stale-while-revalidate `20 / 25`
`getWithFreshness()` returns a `CacheResult` with `CacheFreshness.fresh` or `CacheFreshness.stale`, allowing callers to show cached data immediately while deciding whether to refresh. Phased loading in `ForecastService` uses this well. Deduction: the background refresh completion path is not always smooth — there is no guaranteed UI update when the fresh data arrives.

### 2.5 Deterministic cache keys `4 / 5`
Keys follow `reach_cache_` + `reachId`. Deterministic and collision-free for the current data model. Minor deduction for not having a systematic key strategy across different data types.

### 2.6 Cache size limits & eviction `5 / 10`
`_responseCacheMaxSize = 10` bounds the in-memory response cache. The `SharedPreferences` disk cache is effectively unbounded — `cleanupStaleEntries()` must be called manually and only removes entries older than 6 months. A heavy user who loads hundreds of reaches over time will accumulate megabytes of stale SharedPreferences entries.

### 2.7 Offline-first data access `10 / 25`
Reach data cached in SharedPreferences is loaded on app start. If a user opens a previously viewed reach while offline, the cached data is available. However there is no explicit offline mode: no connectivity check, no "you are viewing cached data from X hours ago" indicator, and no communication to the user that the data may be stale.

## Category 3 — State Management `64 / 100`

### 3.1 Single consistent pattern `15 / 15`
Provider + ChangeNotifier used consistently throughout. GetIt registers singleton services. No mixing with Bloc, Riverpod, or raw `setState`. Full marks.

### 3.2 Explicit loading / error / data states `18 / 25`
`FavoritesProvider` exposes `isLoading`, `isEmpty`, `errorMessage`, and per-reach `isRefreshing(reachId)`. `AuthProvider` exposes `_isLoading` and `_errorMessage`. The three-state model is applied to the main providers. Deduction: not all async operations in the app are guaranteed to model all three states consistently — smaller async calls (settings save, reorder) may silently fail without surfacing an error state.

### 3.3 No business logic in widgets `10 / 20`
The main providers handle data fetching and state. However, without a use case layer, some business logic (flow category calculation, return period comparison, sort order) sits in providers and services that are mixed with state management. Pages call directly into providers rather than through use cases. Some formatting and conditional display logic may exist in widget `build()` methods.

### 3.4 Granular state and selective rebuilds `7 / 15`
`Consumer<FavoritesProvider>` is used in the main favorites page. However `select()` is not used to subscribe to sub-slices of provider state. Any `notifyListeners()` call rebuilds all `Consumer` widgets watching that provider, even if the specific data they display did not change.

### 3.5 Immutable state objects `2 / 5`
`ReachData` uses `copyWith()` ✅. Provider state variables are mutated in-place and then `notifyListeners()` is called. This is the standard ChangeNotifier pattern but means state is not immutable — a listener could observe partially updated state.

### 3.6 Proper resource disposal `12 / 20`
`AuthProvider` cancels its `_authStateSubscription` in `dispose()` ✅. However the FCM `onTokenRefresh` listener in `FCMService` is set up without storing the subscription reference, making it impossible to cancel. Other stream subscriptions in the codebase need a sweep to confirm consistent disposal.

## Category 4 — Error Handling & Resilience `41 / 80`

### 4.1 Typed error hierarchy `7 / 20`
`ApiException(message: String)` is the only custom exception type. `ErrorService` maps `FirebaseAuthException.code`, Firestore codes, and `SocketException` to string messages. The system maps errors to strings, not to typed error objects. The UI cannot conditionally render different widgets based on error type (e.g., show a "sign in again" button specifically for auth errors) without parsing the message string.

### 4.2 User-facing error messages `12 / 20`
`ErrorService` covers Firebase Auth, Firestore, and network error codes with readable messages. `AuthErrorDisplay` surfaces these in the auth flow. Favorites and map errors surface `errorMessage` from providers. Not verified: whether every possible failure path in the app surfaces a message rather than a blank or frozen screen.

### 4.3 Graceful degradation `12 / 20`
`ForecastService.loadSupplementaryData()` wraps return period and medium-range fetches in individual try-catch blocks and continues with partial data:
```dart
} catch (e) {
  AppLogger.warning('ForecastService', 'Return periods failed, continuing: $e');
  // Continue without return periods
}
```
Good isolation at the forecast level. Less certain about cross-feature failures — if Firestore is unavailable and user settings cannot load, the behavior of the rest of the app is unclear.

### 4.4 Global exception boundary `5 / 10`
Both `FlutterError.onError` and `PlatformDispatcher.instance.onError` are configured in `main.dart` ✅. Both route to `ErrorService.logError()` which calls `developer.log()`. This is better than nothing but `developer.log()` output is only visible when the device is connected to a development machine. No crash reporter is attached — production crashes are invisible.

### 4.5 Retry affordance in UI `5 / 10`
`_buildInitErrorState()` in `FavoritesPage` exists, suggesting some error screens have a retry action. Not comprehensively verified across all screens and error states.

## Category 5 — Performance & Optimization `23 / 60`

### 5.1 Pagination / lazy loading `4 / 20`
The favorites list is inherently small and pagination is not relevant there. However Firestore user documents are fetched without `.limit()`. The more significant gap is that the NOAA API is called per-reach with no batching or deduplication at the HTTP level across multiple simultaneous reach loads.

### 5.2 Network image caching `7 / 10`
The app is not network-image-heavy. Map tiles are cached by the Mapbox SDK ✅. User-selected background images are stored locally. There is no `cached_network_image` dependency, which is acceptable given the current image usage pattern.

### 5.3 Widget build optimization `5 / 10`
`const` constructors are presumably used in simple widgets. The forecast widget hierarchy is deep and complex (`forecast_detail_template`, `chart_preview_widget`, `daily_expandable_widget`, etc.) — not verified whether all leaf widgets use `const`. No evidence of `RepaintBoundary` usage for expensive chart repaints.

### 5.4 Cold start performance `5 / 10`
`Firebase.initializeApp()` runs before `runApp()` in `main()`, blocking the first frame. `_initializeServices()` is called in `initState()` with `setState()` on completion, which defers those operations correctly. The service locator setup (`setupServiceLocator()`) also runs before `runApp()`.

### 5.5 Release build optimization `2 / 10`
No `--obfuscate` or `--split-debug-info` flags found in any build configuration. Debug symbols are shipped with release binaries, making the app easier to reverse-engineer.

## Category 6 — Security `45 / 70`

### 6.1 No secrets in source code `25 / 25`
All secrets are gitignored and never committed:
- `lib/core/config.dart` — Mapbox token, NWM API URLs
- `lib/firebase_options.dart` — Firebase keys
- `android/app/google-services.json` — Android Firebase config
- `ios/Runner/GoogleService-Info.plist` — iOS Firebase config
- `functions/.env` — NWM API key (rotated after a previous exposure)

Full marks. Security posture at the secrets level is strong.

### 6.2 Secure storage for sensitive data `18 / 20`
`flutter_secure_storage` is used for auth tokens, biometric preferences, and user credentials. On Android: `EncryptedSharedPreferences`. On iOS: Keychain with `first_unlock_this_device` accessibility. Small deduction: unclear whether all sensitive Firestore-derived data (user email, name) is protected consistently.

### 6.3 Certificate / SSL pinning `0 / 10`
Not implemented. NOAA API and NWM API traffic is not pinned. A compromised network could intercept API responses and serve manipulated flow data to users.

### 6.4 Release obfuscation `0 / 10`
No `--obfuscate` or `--split-debug-info` configuration found. Not configured in any build script or `Makefile`. The release binary ships with readable class/method names.

### 6.5 Input validation at boundaries `2 / 5`
Firebase Auth validates email format server-side. No explicit client-side validation or sanitization found for text inputs. API responses are not validated against a schema before parsing.

## Category 8 — Clean Architecture `14 / 90`

> This is the most critical gap. The score of 14/90 reflects the absence of Clean Architecture principles that every other category depends on.

### 8.1 Three-layer structure enforced `4 / 20`
The actual structure:
```
lib/
  core/
    services/    -- auth, forecast, NOAA API, FCM, cache (all mixed)
    providers/   -- state management
    models/      -- data models (serve as both entities and DTOs)
  features/
    auth/
      presentation/  -- pages and widgets
      providers/     -- auth_provider.dart
    favorites/
      pages/
      services/      -- coach mark, video
      widgets/
    forecast/
      domain/
        entities/    -- ONLY domain layer in the app (partial)
      pages/
      services/      -- daily_forecast_processor
      widgets/
    map/
      models/
      services/
      widgets/
```
Only `features/forecast/domain/entities/` shows Clean Architecture intent. All other features are flat `pages/services/widgets`. Adding a new data source requires touching service files that also manage business logic.

### 8.2 Pure domain layer `2 / 20`
`forecast/domain/entities/daily_flow_forecast.dart` is the only domain entity that avoids framework imports. All other "business objects" (`ReachData`, `UserSettings`, `FavoriteRiver`) import framework packages or are entangled with serialization concerns. Business rules (flow category calculation, return period thresholds) live in `ReachData` and `NoaaApiService` alongside I/O code.

### 8.3 Repository abstraction `5 / 15`
`INoaaApiService` and `IFCMService` are abstract interfaces ✅. These enable mocking and are a step in the right direction. However they are service interfaces, not repository contracts — they expose method signatures tied to the NOAA API shape rather than domain operations. Replacing NOAA with a different data source would require changing the interface and all callers.

### 8.4 Use cases `0 / 20`
No use case classes exist anywhere. Business operations (load forecast, add favorite, send alert) are implemented directly in providers and services. There is no single `call()` entry point per operation. Providers call services directly; there is no layer between them that encapsulates a single business operation.

### 8.5 DTOs separate from domain entities `3 / 15`
`ReachData` serves as both the domain entity and the DTO — it has `fromNoaaApi()`, `fromReturnPeriodApi()`, and `fromJson()` factory constructors alongside domain methods like `getFlowCategory()`. `UserSettings.fromJson(doc.data()!)` parses directly from Firestore in the provider. Changing the NOAA API response schema requires touching the core domain model.

## Category 9 — User Experience & Accessibility `20 / 80`

### 9.1 Skeleton / shimmer loading screens `2 / 15`
All loading states use `CupertinoActivityIndicator` with a text label:
```dart
// lib/features/favorites/favorites_page.dart
Widget _buildLoadingState() {
  return const Center(
    child: Column(
      children: [
        CupertinoActivityIndicator(radius: 16),
        SizedBox(height: 16),
        Text('Loading your rivers...'),
      ],
    ),
  );
}
```
No shimmer or skeleton placeholder cards. Users see a spinner where content will appear, giving no sense of the incoming layout. This is the most visible UX gap on first load.

### 9.2 Optimistic updates `6 / 10`
`FavoritesProvider.addFavorite()` updates local state before the Firestore write completes. The favorite appears in the list immediately. Rollback on failure is unclear. Not verified for all mutation types.

### 9.3 Pull-to-refresh `5 / 10`
Not directly verified in this audit. Common pattern in Cupertino apps; partially credited. Needs confirmation across all data screens.

### 9.4 Empty state screens `7 / 10`
`_buildEmptyState()` exists in `FavoritesPage` with contextual messaging. Map likely has an empty state for no results. Deduction: not verified on forecast or settings screens.

### 9.5 Offline / connectivity banner `0 / 15`
No connectivity detection (`connectivity_plus` not in `pubspec.yaml`). No offline indicator. A user who loses internet while using the app receives no signal — they will see stale data or API error messages with no explanation that they are offline.

### 9.6 Text scale factor support `0 / 15`
Zero occurrences of `textScaleFactor`, `TextScaler`, or `MediaQuery.textScalerOf` found in the codebase. Fixed font sizes are used throughout. At iOS system font size ×1.5 or ×2.0, text will overflow containers, clip, and break layouts. Users who rely on larger system text — a significant portion of an outdoor/safety app's audience — will have a degraded experience.

### 9.7 Accessibility semantics `0 / 5`
Zero `Semantics(` widgets found. No `semanticLabel` on images or icon buttons. Screen reader users cannot navigate primary flows. Cupertino widgets provide some baseline accessibility through their internal semantics, but custom widgets and icon-only buttons have no labels.

## Category 10 — Observability & Monitoring `3 / 40`

### 10.1 Crash reporting `2 / 15`
`FlutterError.onError` and `PlatformDispatcher.instance.onError` are configured and route to `ErrorService.logError()`. This is a correct error boundary. However `ErrorService.logError()` calls `developer.log()`, which is only visible in an IDE debugger. In production with no device attached, all crashes are silent. `firebase_crashlytics` is not in `pubspec.yaml`.

### 10.2 Analytics instrumentation `0 / 15`
`firebase_analytics: ^12.0.0` is in `pubspec.yaml`. Zero analytics events are logged anywhere in the codebase. The package is initialized but produces no data. There is currently no way to know which features users use, what forecasts they view, or where they drop off.

### 10.3 API latency and error rate tracking `1 / 10`
`AppLogger.warning()` logs retry attempts and API failures locally. No remote visibility. No way to know if NOAA or the NWM API is degraded without a user report.

## Category 11 — Routing & Navigation `8 / 20`

### 11.1 Declarative routing `4 / 10`
`AppRouter` centralizes named routes and typed route arguments in one file. Routes are defined via `CupertinoApp.routes`, `onGenerateRoute`, and `onUnknownRoute`. This is better than scattered `Navigator.push` calls. However it is still imperative — no `go_router` or `auto_route`. Deep links, URL-based navigation, and route guards cannot be cleanly attached.

### 11.2 Deep link support `0 / 5`
No deep link handling. Push notification taps route via a `GlobalKey<NavigatorState>` passed to `FCMService` — a workaround rather than a routing solution. An incoming link like `rivr://reach/12345` would not be handled.

### 11.3 Centralized route guards `4 / 5`
`AuthCoordinator` centralizes auth state routing. Unauthenticated users are redirected before they see protected content. Small deduction: the guard is at the widget level rather than inside a declarative router, making it harder to extend as routes grow.

## Category 14 — Firebase Services `29 / 80`

### 14.1 Firestore type-safe access with `.withConverter()` `0 / 15`
Not used. All Firestore reads use manual JSON casting:
```dart
// lib/core/services/user_settings_service.dart
final settings = UserSettings.fromJson(doc.data()!);
```
`doc.data()!` is a nullable assertion on a raw `Map<String, dynamic>`. Firestore schema changes will cause silent runtime failures rather than compile-time errors. `.withConverter<UserSettings>()` would enforce type safety at the SDK level.

### 14.2 Firestore security rules version-controlled `0 / 10`
`firestore.rules` does not exist in the repository. Security rules are managed exclusively via the Firebase Console. They are not code-reviewed, not version-controlled, and not deployed via CLI alongside application changes. A mistake in the Console could expose all user data silently.

### 14.3 Firestore listener lifecycle management `5 / 10`
The app uses one-time `.get()` reads for all Firestore access rather than real-time `snapshots()` listeners. This means there are no dangling listeners to manage (no penalty), but it also means the app has no real-time data capability — user settings or favorites changed from another device will not sync until the app is restarted.

### 14.4 Auth state stream as source of truth `9 / 10`
`AuthProvider.initialize()` subscribes to `FirebaseAuth.instance.authStateChanges()`. The subscription is stored and cancelled in `dispose()`. Auth state changes propagate automatically. Near-perfect implementation.

### 14.5 Auth error codes mapped to user messages `7 / 8`
`ErrorService` maps: `wrong-password`, `user-not-found`, `too-many-requests`, `email-already-in-use`, `weak-password`, `invalid-email`, `user-disabled`, and network-related codes. Comprehensive coverage. Small deduction: the verification email expiry issue (referenced in feedback) suggests at least one auth edge case is not fully handled in the UX.

### 14.6 FCM token refresh handled `5 / 8`
`onTokenRefresh.listen()` is set up in `FcmService`:
```dart
_messaging.onTokenRefresh.listen((newToken) async {
  _cachedToken = newToken;
  await _saveTokenToUserSettings(userId, newToken);
});
```
Token is stored and written to Firestore. Deduction: the subscription is not stored in a variable — it cannot be cancelled when the service is torn down, creating a potential memory leak. The token rotation is only set up inside `refreshTokenIfNeeded()`, meaning it may not be registered on every app start.

### 14.7 Notification permission requested contextually `3 / 5`
Notification permission appears to be requested in the notification settings flow rather than on cold launch. Partially credited — full marks require verifying that iOS provisional permission is used to deliver quiet notifications before the user explicitly grants permission.

### 14.8 Firebase Crashlytics `0 / 8`
`firebase_crashlytics` is not in `pubspec.yaml`. All production crashes are invisible. This is the single highest-priority item to add to the project.

### 14.9 Firebase Analytics actively instrumented `0 / 6`
`firebase_analytics: ^12.0.0` is in `pubspec.yaml`. Zero `logEvent()` calls exist anywhere. The package adds dependency weight but provides no value in its current state. Either instrument it or remove it.
